# HTML-трекер тренировки (Apps Script + Google Sheets)

Мобильное приложение для трекинга тренировки в реальном времени. Читает план дня из
вкладки «План», пишет факт по подходам во вкладку «Сессии» (кардио — в «Кардио»). Хостится самим
Apps Script — отдельный сервер не нужен. Агент (`/plan-next-week`) забирает «Сессии» и планирует дальше.

- **Контракт данных:** `DATA-CONTRACT.md` (структура листов и JSON-обмена)
- **Бэкенд:** `server.ts` (TypeScript) — `doGet` (отдаёт app), `getPlan`, `listPlans`, `saveSession`/`saveCardio`
  (общий идемпотентный `saveRowsIdempotent`, с валидацией). Типы Apps Script — через `@types/google-apps-script`.
  Чистая логика — `logic.ts`. Серверные `.gs` для Apps Script **генерируются** из `.ts` (см. ниже).
- **Фронтенд:** Svelte + Vite, исходники в `ui/src/`, собираются в один `index.html` (inline JS+CSS)
  плагином `vite-plugin-singlefile` — формат, который требует Apps Script. Offline-first, `localStorage`.

`logic.ts` — **единый типизированный источник** правил (ESM): клиент (`ui/src/`) импортирует его напрямую,
Node-тесты импортируют его напрямую (Node strip-types). Серверные `logic.gs`/`server.gs` для Apps Script
**генерируются** из `logic.ts`/`server.ts` скриптом `ui/gen-gs.mjs` (esbuild снимает типы, `import`/`export`
убираются → top-level глобальные функции). `.gs` — генерируемые артефакты (в `.gitignore`), править их руками нельзя.

## Локальная разработка

```bash
cd projects/training/app/ui
npm install
npm run dev      # Vite dev-сервер; google.script.run недоступен → gas() отдаёт mock-план
```

Вне Apps Script (`npm run dev`) `lib/gas.ts` подставляет mock (план/сохранение), чтобы крутить UI локально.
Симуляция офлайна для теста ретраев: в консоли браузера `localStorage.mock_offline = '1'` (вернуть — `'0'`).

## Деплой

Через [`clasp`](https://github.com/google/clasp). Скрипт привязан к Google-таблице (bound script):
`SPREADSHEET_ID` в `deploy.sh` — точка привязки. Для другой таблицы поменяй эту переменную.

### Что нужно от таблицы

В таблице с этим `SPREADSHEET_ID` должны быть:

- хотя бы одна вкладка `План: <название>` (напр. `План: fullbody`);
- лист `Сессии` (15 колонок) и лист `Кардио` (11 колонок) с заголовками по `DATA-CONTRACT.md`.

### Разовая настройка

1. `npx @google/clasp login` — авторизоваться в clasp.
2. Включить Apps Script API: https://script.google.com/home/usersettings.

### Сборка и пуш

```bash
cd projects/training/app
./deploy.sh
```

`deploy.sh`: гоняет тесты `logic.ts` → собирает UI (`ui` → один `index.html`) → **типчек** (`tsc --noEmit`
по `logic.ts`/`server.ts`) → генерирует серверные `logic.gs`+`server.gs` из `.ts` (`ui/gen-gs.mjs`) → проверяет,
что в бандле нет `eval`/`new Function` (CSP песочницы) → `clasp push` (`index.html`, `server.gs`, `logic.gs`,
`appsscript.json`) → обновляет существующее боевое развёртывание (URL `/exec` не меняется).

**Первый запуск** (когда `.clasp.json` ещё нет): `deploy.sh` сам создаёт привязанный к таблице скрипт
(`clasp create-script --parentId $SPREADSHEET_ID`) и делает **первое** развёртывание, печатая URL `/exec`.
После этого один раз открой выданный `/exec` URL **на десктопе** и пройди OAuth-авторизацию (скрипт
запрашивает доступ к таблице) — на телефоне форму согласия проходить неудобно.

Доступ веб-приложения — **«Только я»** (`appsscript.json`: `access: MYSELF`), запуск **от лица открывающего**
(`executeAs: USER_ACCESSING`). Анонимный доступ закрыт.

`index.html` в корне — **сгенерированный** артефакт сборки (в `.gitignore`); source — это `ui/src/`.
Серверные `logic.gs`/`server.gs` тоже **генерируются** (из `logic.ts`/`server.ts`) и лежат в `.gitignore`.

## Секрет записи (опционально)

По умолчанию запись открыта (защита — только настройкой доступа веб-приложения, «Только я»).
Чтобы потребовать секрет:

1. Apps Script → **Настройки проекта → Свойства скрипта** → добавь `SHARED_SECRET` = твоя строка.
2. В приложении нажми **⚙** и впиши тот же секрет (хранится в `localStorage` телефона).

После этого запись без верного секрета отклоняется (`{ok:false, code:"forbidden"}`).

## Использование

1. Агент/ты заполняет вкладку «План» (формат — `DATA-CONTRACT.md`).
2. Открываешь приложение на телефоне → на старте короткий **экран загрузки** (бренд + прогресс), пока тянется план.
3. Слева в шапке — крупный **селект-дропдаун НЕДЕЛИ** (по умолч. последняя неделя); справа — кнопки финиша (флаг) и настроек **⚙**. Какой именно день открыт — видно по **активной вкладке** (дни — вкладки сверху). **Шапка липкая** (селект недели + вкладки + статус не скроллятся; прокручивается только лента контента под ними). **Неделя** выбирается прямо в **шапке** — смена мгновенно перестраивает дни/подходы (из кэша, без рефетча). В **настройках ⚙** — только **План**, секрет записи и звук сигнала отдыха; недели там больше нет. Смена плана перечитывает таблицу (с экраном загрузки) и обновляет список недель в шапке.
4. По каждому подходу вписываешь Повторы / Вес / RPE, тапаешь ✓. 🔥-разминки приходят из плана. Под номером (#) видна длительность отдыха; **тап по # переключает тип подхода** — разминка (🔥) ↔ рабочий (таймер он НЕ запускает).
5. **Таймер отдыха** авто-стартует при ✓ любого подхода — рабочего или разминки (длительность — из колонки «Отдых» плана, иначе дефолт). Док снизу: −15 / пауза / +15 / «Пропустить». Если отдых превышен — счёт идёт вверх янтарным. Таймер переживает блокировку экрана и перезагрузку (моменты по `Date.now()`).
6. Степпер **−/+** под подходами — менять число подходов. Кардио-день — форма (вид/мин/пульс/RPE).
7. **Завершить** — синяя кнопка-флаг в **шапке** (рядом с ⚙). Пока отправлять нечего (силовая без единого ✓/повтора или кардио без длительности) — кнопка **неактивна**. При нажатии (силовая — с оценкой самочувствия 1–10) данные один раз уходят в «Сессии» или «Кардио».
   - В черновике (пока копишь подходы) **статус-строка скрыта** — лишней плашки нет. Она появляется под вкладками только при отправке (**«Отправляем…»** → **«Отправлено»**) или офлайне (**«Оффлайн — отправится позже»** с кнопкой **↻ Повторить** прямо в строке).
   - Завершённый день не блокирует переключение на другой — он начнётся свежей сессией.
   - Нет сети — всё сохранено локально; ретрай-кнопка **↻** живёт в статус-строке (авто-ретрай при возврате сети).
   - Повторная синхронизация той же тренировки не создаёт дублей (идемпотентность по `session_id`).

## Тесты

Чистая логика (`logic.ts`) покрыта юнит-тестами на встроенном раннере Node (без зависимостей).
Тесты импортируют `logic.ts`/`timer.ts` напрямую как ESM (Node strip-types — нужен **Node ≥ 23.6**):

```bash
cd projects/training/app
npm test          # node --test tests/*.test.js
```

Тесты проверяют: парсинг плана (вкл. колонку «Отдых» — оба формата, backward-compat), валидацию payload
(коды ошибок, диапазоны RPE/вес/повторы, лимит размера), идемпотентность (`findDuplicate`), сборку строк
«Сессии»/«Кардио», сбор подходов из состояния (`collectSets`). Отдельный набор — **rest-таймер**
(`tests/timer.test.js`, чистый модуль `ui/src/lib/timer.ts` на фейковых часах: старт/пауза/±/пропуск/переотдых,
serialize/restore после reload). `server.ts` и клиент вызывают те же функции из `logic.ts` — тесты покрывают
и серверную, и клиентскую логику.

**Юнит-тесты UI (Vitest + jsdom)** — стор, чистые трансформы состояния и серверная идемпотентность:

```bash
cd projects/training/app/ui
npm test          # vitest run (src/lib/*.test.ts, tests/*.test.ts)
```

Покрывают `store` через фасад (`loadPlan`/`touch`/`syncNow` online+offline/guard смены дня-недели/`startNew`/
`syncStatus`/сбой localStorage), `state` (`buildState`/`buildPayload`, фильтр по неделе, rest-дефолты) и
`saveRowsIdempotent` с моками Apps Script-глобалов (forbidden/busy/no_sheet/duplicate/success). Стор разнесён
на доменные модули `ui/src/lib/store/` (`appState`/`loading`/`restTimer`/`session`/`plan`/`sync`) за фасадом
`store.svelte.ts` — компоненты импортируют только фасад.

**E2E (Playwright)** — поверх локального Vite dev-сервера с offline-mock бэкендом (`ui/src/lib/gas.ts`):

```bash
cd projects/training/app/ui
npm run e2e       # playwright test (e2e/) — при первом запуске: npx playwright install chromium
```

Покрывают пользовательские сценарии на реальном DOM: загрузка плана/вкладки/заметки, лог→завершение→lock,
переключение дня (в т.ч. после завершения), rest-таймер (старт/±/пауза/пропуск), смена недели в шапке,
кардио-форма, restore после reload, offline→ретрай. Селекторы — стабильные (`data-testid`/`aria`).

Проверка типов: серверная (`logic.ts` + `server.ts`, strict, `@types/google-apps-script`) — `cd ui && npm run typecheck`
(`tsc --noEmit -p tsconfig.gs.json`, тот же типчек выполняет `deploy.sh` перед генерацией `.gs`); клиентская
(весь `ui/src`, strict) — `cd ui && npm run check` (`svelte-check`, 0 ошибок); тесты (vitest, тянут `server.ts`
с GAS-глобалами) — `cd ui && npm run test:typecheck` (`svelte-check` по `tsconfig.test.json`).

## Ограничения прототипа

- **Offline — «resilient, не cold-start».** Открой приложение при наличии сети (план кэшируется);
  оно переживёт потерю связи во время тренировки и синканёт в конце. Полностью офлайн «с нуля»
  Apps Script-хостинг не гарантирует — это отдельный шаг (PWA).
- Источник правды по структуре — `DATA-CONTRACT.md`. Менять колонки — синхронно здесь и в коде.
- Вкладку «Активная» приложение не трогает.
